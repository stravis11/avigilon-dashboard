What is the Avigilon Control Center™ Software?
The Avigilon Control Center (ACC) video management software allows you to record and view
live video from multiple video security cameras, monitor alarms, control cameras, and archive
video. Video security teams can use the ACC software for live monitoring and use the
archived footage for investigative and forensic purposes. The ACC software gives users
control over their workstation with a simplified screen layout, intuitive controls, and
customizable features that enable multiple user interactions.
The ACC Server software is the application that captures and records video security data from
network cameras and encoders. The captured data is sent to the ACC Client software for
security personnel to review. The ACC Client software works with the ACC Server software to
give you access and control of your video security system. The ACC Client software allows
you to view live and recorded video, monitor events, and control user access to the ACC
system. The ACC Client software also gives you the ability to configure your video security
system.

Editions
There are 3 editions of the ACC software: Core, Standard, and Enterprise.

simultaneous ACC Client connections, and which features are available in the ACC Client

Sites and Servers
In the ACC system, servers and connected devices are organized into groups called sites. By
organizing the system into sites, you can control user access and system-wide events
through the site settings. The site settings are stored on the server, or across all servers in a
multi-server system.

Within a site, individual servers are responsible for managing the devices that are connected
to it. Specifically, the server controls video recording. Using the server settings, you can
control when video is recorded, how long it is stored, and how much bandwidth is used to

specific alarms in the ACC system.

When planning your system integrations, refer to the following best practices in your
integration software:
• Create a user with restricted or non-administrator access.
• Create a logout mechanism to prevent unlimited number of sessions or connections.
• Create a session timeout mechanism for idle sessions.
• Monitor the TCP connections to ACC to prevent CPU overload on ACC software.

generating hundreds of view panels.
• Alarms must either be acknowledged or purged after a fixed maximum number of
unacknowledged alarms remain pending.

You can set up webhooks with WEP to receive events from an ACC site. Webhooks are

• A valid url that will receive ACC webhook notifications.
• A list of eventTopics that you would like to subscribe to. For more information, see

• An authenticationToken to verify event messages are originating from the ACC
webhook service.

Webhooks vs Socket.IO
Both Socket.IO and webhooks are used for real time updates between ACC servers and
clients. Webhooks are recommended as they can provide notifications for the whole site,
whereas Socket.IO only provides notifications from the server that the WEP service is running
on. The WEP service and the Socket.IO connection need to be running on each server within
the site.

Webhooks
Webhooks provide site-wide notifications and are recommended over Socket.IO.
Webhooks are callbacks that are triggered by events on the ACC Server. The client can
configure a public URL to receive events from an ACC site. Webhooks are available over
HTTP/HTTPS protocol and are configured for site-wide communication. Optional heartbeat
notifications can be configured to make sure connection to the WEP service has not been
interrupted.

Pros
1. WEP forwards notifications via a non-persistent HTTP/HTTPS connection.
2. Performs well across a multi-server site.
3. Server and site-wide notifications are sent to the client.

Cons
1. Only events are available via Webhooks. To receive ALARM_OBJECTS ,
CAMERA_OBJECTS , SERVER_OBJECTS and SITE notifications, you will need to
subscribe via Socket.IO.
2. Clients need to setup a receiving HTTP/HTTPS server to make use of this functionality.
3. Each push to HTTP/HTTPS server initiates a new TCP connection.
4. Webhooks can only facilitate one-way communication from server to client.

©2026, Avigilon Corporation. All right reserved.

66

Development and Deployment

Socket.IO
Socket.IO only provides notifications from the server that the WEP service is running on and is
not recommended over webhooks.
Socket notifications rely on persistent web socket connections between the third party client
and WEP, and a notification channel between WEP and the ACC server. The socket-based
connection is configured between a specific server and a client.

Pros
1. Fast two-way channel between a web browser and a server.
2. Low latency communication between a client and web service.
3. ALARM_OBJECTS , CAMERA_OBJECTS , SERVER_OBJECTS , SITE notifications, and
events are available through socket connection.

Cons
1. Session required for full life cycle: A session is required to open the notification channel
as well as to keep the notification channel alive. These sessions can expire, which means
that the client would have additional work to regenerate a new token and resubscribe to
notifications again.
2. The client has additional responsibility to listen to the notification channel disconnect
event and re-establish the notification channel subscription if the connection goes down
for some reason.
3. WEP needs to be installed on each server and Socket.IO connection is required for each
server to receive site-wide events.
4. Only server-specific notifications are sent to the client.
5. Performance and Scale : Due to multiple channels for each server, with multiple clients,
we have exponential channels open between the ACC server and the WEP service.

/camera/commands/pan-tilt-zoom
PUT /camera/commands/pan-tilt-zoom
Trigger a continuous or field-of-view PTZ action
A PTZ action can be triggered by specifying either the continuous mode or the
fieldOfView mode.
Continuous PTZ
The continuous mode triggers a continuous PTZ action with a joystick, mouse-drag, or
zooming in or out.
You can call multiple START commands with varying values for panAmount , tiltAmount
and zoomAmount . The last call will continue until another START or STOP command is
received. Specify a STOP command when the mouse button or joystick is released. If there is
no STOP command, the last call will continue until the end of the camera range is reached.
This mode requires visual verification of the video stream. Use the camera web UI or the ACC
client to view the stream while calling a command.
Field-of-View PTZ
The fieldOfView mode changes the camera field-of-view (FOV). Only one of the following
two modes can be specified at a time:
• ROI — zooms into a sub-region of the current camera FOV.
• point — centers the FOV on a specified point within the current camera FOV.
For both ROI and point values, any unit can be used as long as it is consistent. The API
normalizes the units.

Web Endpoint API Routes

PUT /recording/retention
Update the video retention settings
WARNING
Updating these settings can result in data loss.
This call updates the retention settings for cameras connected to a server. The request only
applies to the cameraIds entered.
Specify the maximum amount of time to keep recorded video on the server using the
maxRetention parameter. Video older than this setting is deleted from the server, unless it
is part of a protected bookmark. If the server runs out of space before this time, the oldest
unprotected video will be deleted first.
To increase storage capacity, the ACC system can delete a portion of the higher resolution
streams using the streamRetentionAmount parameter. To use this parameter, the camera
must have a secondary stream enabled.
The streamRetentionAmount parameter is an array with two values, [x, y] :
• For JPEG cameras with 3 streams, specify the percent of video kept at full frame rate (x)
and half frame rate (x + y) . The ACC system keeps 100% of the quarter frame rate until
it exceeds the maxRetention parameter.
• For H.264 and H.265 cameras with 2 streams, specify the percent of video kept at high
resolution (x + y) . Both x and y values are required in case the camera format
changes to JPEG. The ACC system keeps 100% of the low resolution stream until it exceeds
the maxRetention parameter.
For example,
• for JPEG cameras, [0.6, 0.2] would indicate 60% full frame rate, 80% half frame rate
and 100% quarter frame rate.
• for H.264 cameras, [0.6, 0.2] would indicate 80% high resolution and 100% low
resolution.
